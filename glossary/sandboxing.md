# Sandboxing (การทำแซนด์บ็อกซ์)

[← กลับสู่เนื้อหาบทเรียน](../lectures/project3_agentic_workflows_tools.md)

**Sandboxing (แซนด์บ็อกซ์)** เป็นกลไกด้านความปลอดภัยของซอฟต์แวร์ที่ใช้ในการ "กักบริเวณ" หรือจำกัดขอบเขตพื้นที่ให้โปรแกรม โค้ดคอมพิวเตอร์ หรือ Agent ทำงานอยู่ในสภาพแวดล้อมที่ถูกแยกออกไปอย่างโดดเดี่ยว (Isolated environment) โดยไม่สามารถไปยุ่มย่ามหรือทำความเสียหายให้กับระบบปฏิบัติการหลัก (Host OS) หรือข้อมูลส่วนตัวของผู้ใช้ได้

เปรียบเสมือนการให้เด็กเล่นอยู่ในกระบะทราย (Sandbox) ซึ่งเด็กจะทำทรายหกเลอะเทอะหรือสร้างปราสาททรายอย่างไรก็ได้ แต่อนุญาตให้เล่นได้แค่ในขอบเขตกระบะทรายเท่านั้น

## คำอธิบายเชิงลึก (In-depth Explanation)

เมื่อเราใช้งาน **Autonomous AI Agents** ที่สามารถเขียนโค้ดและรันโค้ดได้เอง การทำแซนด์บ็อกซ์เป็นสิ่งที่ **หลีกเลี่ยงไม่ได้** เพราะ AI มีโอกาสที่จะสร้างโค้ดที่ผิดพลาด (Bug) ทำให้เครื่องค้าง ทำลายไฟล์ระบบโดยไม่ได้ตั้งใจ (เช่น สั่ง `rm -rf /`) หรือแม้แต่สร้างช่องโหว่ด้านความปลอดภัย

เทคโนโลยี Sandboxing มักทำผ่านกลไก 3 ระดับ:

1.  **Virtual Machines (VM)**: ระบบจำลองคอมพิวเตอร์ขึ้นมาทั้งเครื่อง มี OS ของตัวเอง (ช้า ปลอดภัยสุด)
2.  **Containers (เช่น Docker)**: แบ่งปัน OS Kernel กัน แต่แยกพื้นที่ File System, Network, Process ออกจากกัน (สมดุลระหว่างความเร็วและความปลอดภัย เป็นที่นิยมที่สุด)
3.  **Process-level Sandboxing (เช่น eBPF, seccomp)**: สกัดกั้นคำสั่ง System Call ระดับต่ำที่อันตราย ไม่ให้ส่งไปถึง Kernel

## ตัวอย่างการนำไปใช้ (Practical Example)

- **ChatGPT Code Interpreter (Advanced Data Analysis)**: เมื่อคุณสั่งให้ ChatGPT เขียน Python โค้ดนั้นไม่ได้รันบนเครื่องของคุณ และไม่ได้รันบนเซิร์ฟเวอร์หลักของ OpenAI แต่จะถูกส่งไปรันใน **Docker Container Sandbox** ชั่วคราว เมื่อรันเสร็จหรือหมดเวลา Sandbox นั้นก็จะถูกทำลายทิ้งทันที
- **เว็บเบราว์เซอร์**: แท็บเบราว์เซอร์อย่าง Chrome ใช้กลไก Sandboxing ในการรัน JavaScript หากเว็บหนึ่งมีมัลแวร์ฝังอยู่ มัลแวร์นั้นก็จะติดอยู่ในกรอบของแท็บนั้น ไม่สามารถอ่านไฟล์รหัสผ่านในไดรฟ์ C ของคุณได้
